# Runbook: внешний API возвращает 401/403, но данные нужны

## Когда применять

Используй этот сценарий, если ingest-пайплайн технически работает, но внешний HTTP API стабильно отвечает `401` или
`403`, а публичная HTML-страница того же источника доступна и содержит нужные карточки/объекты.

Признаки:

- scheduler/worker живые, task доходит до HTTP-запроса;
- ошибка повторяется на первой странице API;
- retry не меняет результат;
- HTML-страница поиска/каталога открывается с `200`;
- данные нужны для bronze/raw-слоя, а не для строгого контрактного API.

## Решение

Не маскировать ошибку зелёным `skip`, если бизнес-цель — получить данные. Вместо этого:

1. оставить API основным источником;
2. при `401/403` переключаться на HTML fallback;
3. парсить только устойчивый минимум полей;
4. сохранять результат в bronze как raw JSON с явной меткой источника;
5. ограничить число страниц, паузы и параллелизм.

Если карточка содержит ссылку на detail-страницу и downstream-слоям нужны расширенные поля, добавь отдельный
ограниченный detail fetch. Он должен обогащать raw payload, но не ломать весь ingest при ошибке одной страницы.
Если detail-страница возвращает captcha или challenge, помечай это отдельным статусом и используй данные карточки как
fallback.

## Минимальный контракт fallback

В raw payload обязательно положить:

- `source_mode`: например `html_fallback`;
- стабильный внешний ID объекта, если его можно извлечь из URL;
- заголовок/название;
- URL исходной страницы;
- ключевые видимые атрибуты карточки;
- полный или усечённый текст карточки;
- поисковые параметры;
- номер страницы выдачи.
- статус detail-догрузки, если она включена.
- значения, извлечённые из карточки, которые могут заменить detail-поля при captcha/challenge.

Так downstream-слои смогут отличать API JSON от HTML-derived JSON и не делать вид, что формат одинаковый.

## Ограничители безопасности

- Не делать бесконечный crawl.
- Не увеличивать concurrency без причины.
- Не ретраить `401/403` как transient-ошибку.
- Не удалять старые failed runs ради косметики.
- Не считать зелёный run успешным ingest'ом без проверки количества строк в хранилище.

Рекомендуемые настройки:

```env
HTML_FALLBACK_MAX_PAGES=5
REQUEST_TIMEOUT=30
INSERT_BATCH_SIZE=500
FETCH_DETAILS=true
DETAIL_MAX_PER_GROUP=100
DETAIL_PAUSE_MIN_SEC=0.2
DETAIL_PAUSE_MAX_SEC=0.6
```

## Проверка результата

После запуска проверь не только статус оркестратора, но и факт появления строк:

```sql
SELECT count(), uniqExact(snapshot_id), max(ingested_at)
FROM bronze_table;

SELECT source_mode, count()
FROM bronze_table
GROUP BY source_mode;
```

Если `source_mode` лежит внутри JSON-поля, используй JSON extraction функции конкретной БД.

## Что документировать

В проектной документации укажи:

- какой endpoint был основным;
- при каких статусах включается fallback;
- какие поля сохраняются из HTML;
- где настраивается лимит страниц;
- какие ограничения есть у HTML-derived данных.

В обезличенном runbook не указывай конкретные домены, имена клиентов, внутренние пути, токены и приватные детали
инфраструктуры.
